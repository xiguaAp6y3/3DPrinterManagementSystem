# 登录认证系统完善路线图

创建时间：2026-07-08  
适用范围：3DPMS 后端与 SQL Server 数据库  
目标阶段：阶段 1 可落地认证系统

## 1. 当前状态

当前项目已经具备认证骨架：

- `POST /api/v1/app/auth/login`
- `POST /api/v1/admin/auth/login`
- `Authorization: Bearer <token>`
- `require_app_user`
- `require_admin`
- JWT 生成与解析
- `users` 表
- `staff_users` 表

但当前还不是可用的真实登录系统：

- APP 登录没有真实验证码校验。
- 管理员登录没有查询数据库。
- 管理员密码没有真实校验。
- JWT payload 没有稳定的 `user_id` / `staff_user_id`。
- 没有刷新 token。
- 没有退出登录。
- 没有登录日志。
- 没有失败次数限制。
- 没有禁用账号后的 token 失效策略。
- 管理端权限只有 `role` 字符串，没有明确权限边界。

## 2. 设计目标

阶段 1 的认证系统不追求复杂，但必须做到：

- 管理员账号密码真实登录。
- 客户 APP 手机号验证码登录，阶段 1 可先用开发验证码或模拟验证码。
- JWT 中包含稳定身份字段。
- 所有业务接口能拿到当前用户 ID 或管理员 ID。
- 支持账号禁用。
- 支持退出登录。
- 支持刷新 token。
- 记录登录成功/失败日志。
- 管理端有基础角色区分。
- 密码使用安全哈希，不存明文。

暂不做：

- 第三方 OAuth。
- 微信小程序登录。
- 复杂多租户。
- 精细到按钮级的 RBAC。
- 在线短信服务强依赖。
- 单点登录 SSO。

## 3. 推荐认证模型

### 3.1 Token 类型

建议使用双 token：

```text
access_token：短期访问 token
refresh_token：长期刷新 token
```

建议有效期：

```text
access_token：2 小时
refresh_token：7 天
```

开发阶段也可以临时使用：

```text
access_token：24 小时
refresh_token：30 天
```

### 3.2 JWT Payload

客户 token：

```json
{
  "sub": "user:123",
  "type": "app",
  "user_id": 123,
  "phone": "13800000000",
  "jti": "uuid",
  "exp": 1234567890
}
```

管理员 token：

```json
{
  "sub": "staff:1",
  "type": "admin",
  "staff_user_id": 1,
  "username": "admin",
  "role": "admin",
  "jti": "uuid",
  "exp": 1234567890
}
```

原则：

- 不再用手机号或用户名直接当唯一身份来源。
- 业务接口从 token 中读取 `user_id` 或 `staff_user_id`。
- 每个 token 都带 `jti`，便于退出登录、刷新 token、审计。

## 4. 数据库表完善路线

### 4.1 保留现有表

现有 `users` 表继续作为客户账号主表。

建议字段保持：

```text
id
phone
nickname
avatar_url
status
created_at
updated_at
```

现有 `staff_users` 表继续作为管理端账号主表。

建议补充字段：

```text
password_changed_at DATETIME2(3) NULL
failed_login_count INT NOT NULL DEFAULT 0
locked_until DATETIME2(3) NULL
last_login_ip NVARCHAR(64) NULL
last_login_user_agent NVARCHAR(500) NULL
```

说明：

- `password_changed_at` 用于密码修改后让旧 token 失效。
- `failed_login_count` 和 `locked_until` 用于防暴力破解。
- `last_login_ip` 和 `last_login_user_agent` 用于后台安全审计。

### 4.2 新增 `auth_refresh_tokens`

用途：

- 保存 refresh token 的哈希。
- 支持退出登录。
- 支持刷新 token 轮换。
- 支持禁用某个设备会话。

建议字段：

```sql
CREATE TABLE dbo.auth_refresh_tokens (
    id BIGINT IDENTITY(1,1) NOT NULL PRIMARY KEY,
    owner_type NVARCHAR(50) NOT NULL,
    owner_id BIGINT NOT NULL,
    token_hash NVARCHAR(128) NOT NULL,
    jti NVARCHAR(100) NOT NULL,
    device_id NVARCHAR(100) NULL,
    device_name NVARCHAR(200) NULL,
    ip_address NVARCHAR(64) NULL,
    user_agent NVARCHAR(500) NULL,
    expires_at DATETIME2(3) NOT NULL,
    revoked_at DATETIME2(3) NULL,
    revoke_reason NVARCHAR(200) NULL,
    created_at DATETIME2(3) NOT NULL CONSTRAINT DF_auth_refresh_tokens_created_at DEFAULT SYSUTCDATETIME(),
    CONSTRAINT UQ_auth_refresh_tokens_jti UNIQUE (jti),
    CONSTRAINT CK_auth_refresh_tokens_owner_type CHECK (owner_type IN (N'app_user', N'staff_user'))
);
```

索引：

```sql
CREATE INDEX IX_auth_refresh_tokens_owner ON dbo.auth_refresh_tokens(owner_type, owner_id, revoked_at, expires_at);
CREATE INDEX IX_auth_refresh_tokens_hash ON dbo.auth_refresh_tokens(token_hash);
```

注意：

- 数据库只保存 refresh token hash，不保存明文 refresh token。
- access token 不落库，除非后续需要强制黑名单。

### 4.3 新增 `auth_login_logs`

用途：

- 记录登录成功/失败。
- 支持安全审计。
- 支持排查账号异常。

建议字段：

```sql
CREATE TABLE dbo.auth_login_logs (
    id BIGINT IDENTITY(1,1) NOT NULL PRIMARY KEY,
    login_type NVARCHAR(50) NOT NULL,
    identifier NVARCHAR(100) NOT NULL,
    owner_type NVARCHAR(50) NULL,
    owner_id BIGINT NULL,
    success BIT NOT NULL,
    failure_reason NVARCHAR(200) NULL,
    ip_address NVARCHAR(64) NULL,
    user_agent NVARCHAR(500) NULL,
    created_at DATETIME2(3) NOT NULL CONSTRAINT DF_auth_login_logs_created_at DEFAULT SYSUTCDATETIME(),
    CONSTRAINT CK_auth_login_logs_login_type CHECK (login_type IN (N'app_sms', N'admin_password')),
    CONSTRAINT CK_auth_login_logs_owner_type CHECK (owner_type IS NULL OR owner_type IN (N'app_user', N'staff_user'))
);
```

索引：

```sql
CREATE INDEX IX_auth_login_logs_identifier_created ON dbo.auth_login_logs(identifier, created_at DESC);
CREATE INDEX IX_auth_login_logs_owner_created ON dbo.auth_login_logs(owner_type, owner_id, created_at DESC);
```

### 4.4 新增 `auth_verification_codes`

用途：

- 阶段 1 模拟或真实短信验证码。
- 防止验证码复用。
- 控制验证码过期时间。

建议字段：

```sql
CREATE TABLE dbo.auth_verification_codes (
    id BIGINT IDENTITY(1,1) NOT NULL PRIMARY KEY,
    phone NVARCHAR(30) NOT NULL,
    purpose NVARCHAR(50) NOT NULL,
    code_hash NVARCHAR(128) NOT NULL,
    expires_at DATETIME2(3) NOT NULL,
    consumed_at DATETIME2(3) NULL,
    send_ip NVARCHAR(64) NULL,
    verify_attempts INT NOT NULL CONSTRAINT DF_auth_verification_codes_attempts DEFAULT 0,
    created_at DATETIME2(3) NOT NULL CONSTRAINT DF_auth_verification_codes_created_at DEFAULT SYSUTCDATETIME(),
    CONSTRAINT CK_auth_verification_codes_purpose CHECK (purpose IN (N'app_login', N'phone_bind'))
);
```

索引：

```sql
CREATE INDEX IX_auth_verification_codes_phone_created ON dbo.auth_verification_codes(phone, created_at DESC);
```

阶段 1 简化策略：

- 可以先不接短信供应商。
- 开发环境允许固定验证码，例如 `123456`。
- 生产环境必须使用 `auth_verification_codes` 校验。

### 4.5 可选新增 `staff_roles` 与 `staff_role_permissions`

阶段 1 可以先不做复杂 RBAC，只保留 `staff_users.role`。

如果后续需要权限细分，再新增：

```text
staff_roles
staff_permissions
staff_role_permissions
staff_user_roles
```

推荐阶段 1 先使用固定角色：

```text
admin       系统管理员
operator    运营/商品/排期
finance     收款确认
reviewer    定制审核/报价
viewer      只读
```

## 5. API 完善路线

### 5.1 APP 端认证接口

#### 发送验证码

```text
POST /api/v1/app/auth/send-code
```

请求：

```json
{
  "phone": "13800000000",
  "purpose": "app_login"
}
```

响应：

```json
{
  "cooldown_seconds": 60,
  "expires_in_seconds": 300
}
```

阶段 1 可选：

- 如果暂不接短信，可以先只记录验证码 hash。
- 开发环境可返回 `debug_code`，生产环境禁止返回。

#### 手机号验证码登录

```text
POST /api/v1/app/auth/login
```

请求：

```json
{
  "phone": "13800000000",
  "code": "123456",
  "device_id": "device-uuid",
  "device_name": "iPhone 15"
}
```

响应：

```json
{
  "access_token": "jwt",
  "refresh_token": "random-secret",
  "token_type": "Bearer",
  "expires_in_seconds": 7200,
  "user": {
    "id": 1,
    "phone": "13800000000",
    "nickname": "用户"
  }
}
```

规则：

- 验证验证码。
- 如果手机号不存在，自动创建 `users`。
- 如果用户 `status = disabled`，返回 `403 USER_DISABLED`。
- 写入 `auth_login_logs`。
- 创建 refresh token 记录。

#### 获取当前客户

```text
GET /api/v1/app/auth/me
```

作用：

- APP 启动后校验 token。
- 返回当前用户基础信息。

#### 刷新 token

```text
POST /api/v1/app/auth/refresh
```

请求：

```json
{
  "refresh_token": "random-secret"
}
```

规则：

- 校验 refresh token hash。
- 校验未过期、未撤销。
- 生成新的 access token。
- 建议轮换 refresh token。

#### 退出登录

```text
POST /api/v1/app/auth/logout
```

规则：

- 撤销当前 refresh token。
- access token 到期前仍可能有效；如果需要立即失效，可引入 access token blacklist。

### 5.2 管理端认证接口

#### 管理员登录

```text
POST /api/v1/admin/auth/login
```

请求：

```json
{
  "username": "admin",
  "password": "password",
  "device_id": "browser-uuid",
  "device_name": "Chrome on Windows"
}
```

规则：

- 查询 `staff_users.username`。
- 校验 `status = active`。
- 校验 `locked_until`。
- 使用 `pwdlib[argon2]` 校验密码。
- 登录失败增加 `failed_login_count`。
- 连续失败 5 次锁定 15 分钟。
- 登录成功清零失败次数。
- 更新 `last_login_at`、`last_login_ip`、`last_login_user_agent`。
- 写入 `auth_login_logs`。
- 创建 refresh token。

#### 获取当前管理员

```text
GET /api/v1/admin/auth/me
```

返回：

```json
{
  "id": 1,
  "username": "admin",
  "display_name": "管理员",
  "role": "admin",
  "permissions": []
}
```

#### 刷新 token

```text
POST /api/v1/admin/auth/refresh
```

#### 退出登录

```text
POST /api/v1/admin/auth/logout
```

#### 修改当前管理员密码

```text
POST /api/v1/admin/auth/change-password
```

请求：

```json
{
  "old_password": "old",
  "new_password": "new"
}
```

规则：

- 校验旧密码。
- 新密码长度至少 10。
- 更新 `password_hash`。
- 更新 `password_changed_at`。
- 撤销该管理员所有 refresh token。

## 6. 服务层落地路线

### 6.1 新增服务

建议新增：

```text
backend/app/services/auth_service.py
```

职责：

- 验证管理员密码。
- 验证 APP 验证码。
- 创建 access token。
- 创建 refresh token。
- 刷新 token。
- 撤销 token。
- 写登录日志。
- 锁定账号。

### 6.2 新增工具函数

建议在 `security.py` 中补：

```text
create_access_token(...)
create_refresh_token_secret()
hash_token(token)
verify_password(...)
hash_password(...)
decode_token(...)
```

注意：

- refresh token 明文只返回给客户端一次。
- 数据库存 `sha256(refresh_token)` 或 HMAC hash。

### 6.3 当前依赖可继续使用

当前 `requirements.txt` 已包含：

```text
PyJWT
pwdlib[argon2]
```

建议继续使用：

- `pwdlib[argon2]` 存管理员密码。
- `PyJWT` 生成 access token。
- Python `secrets.token_urlsafe(48)` 生成 refresh token。

## 8. 阶段验收标准

### 8.1 管理端登录验收

1. 数据库中创建管理员。
2. 使用错误密码登录返回 `401 AUTH_INVALID_CREDENTIALS`。
3. 连续错误 5 次后返回 `423 ACCOUNT_LOCKED` 或 `403 ACCOUNT_LOCKED`。
4. 正确密码登录返回 access token 和 refresh token。
5. 使用 access token 调用 `/api/v1/admin/auth/me` 成功。
6. 禁用管理员后，旧 token 访问业务接口返回 `403 STAFF_DISABLED`。
7. 修改密码后，旧 refresh token 失效。

### 8.2 APP 登录验收

1. 发送验证码成功。
2. 错误验证码登录失败。
3. 正确验证码登录成功。
4. 新手机号自动创建用户。
5. 禁用用户后，旧 token 访问业务接口返回 `403 USER_DISABLED`。
6. refresh token 可刷新 access token。
7. logout 后 refresh token 不能再次使用。

### 8.3 安全审计验收

1. 登录成功写 `auth_login_logs`。
2. 登录失败写 `auth_login_logs`。
3. refresh token 落库时只保存 hash。
4. 退出登录写 `revoked_at`。
5. 后台可以按账号查询登录历史。

## 9. 推荐开发优先级

### P0：必须先做

1. 管理员真实密码登录。
2. APP 手机号登录真实查库或自动建用户。
3. JWT payload 加入 `user_id` / `staff_user_id`。
4. `require_app_user` / `require_admin` 查库确认账号状态。
5. 登录日志。

### P1：建议紧跟

1. refresh token 表。
2. logout。
3. token refresh。
4. 管理员修改密码。
5. 登录失败锁定。
6. APP 验证码表。

### P2：后续增强

1. 精细 RBAC。
2. 设备管理。
3. 管理员强制下线。
4. access token blacklist。
5. 短信供应商接入。
6. 登录地异常提醒。

## 10. 最小可行版本

如果要尽快进入真实后端联调，建议最小版本这样做：

1. 先只实现管理员账号密码真实登录。
2. APP 登录先使用固定验证码 `123456`，但仍真实创建/查询 `users`。
3. access token 有效期 24 小时。
4. refresh token 暂缓，但预留接口和表设计。
5. 所有业务接口通过 token 中的 `user_id` / `staff_user_id` 识别身份。
6. 登录成功/失败必须写日志。
7. 禁用账号必须立即阻止新请求。

这条路线最稳：先解决“谁在操作系统”的问题，再逐步补强“会话如何长期保持、安全如何增强”。
