# 仓库管理与账号体系扩展路线图

## 1. 目标

本路线图用于补齐当前后端缺口：

- 手机 APP 使用邮箱 + 密码登录。
- 手机 APP 支持客户自行注册账户。
- 后台管理员可以增删改客户账户。
- 后台管理员可以增删改内部管理员账户。
- 商品生产完成后，可以选择转移入库。
- 客户订单下所有商品生产并入库后，订单自动变为待发货。
- 一个订单可以填写一个或多个快递单号。
- 填写快递单后，可以批量出库管理。

本路线图不要求接入第三方支付和第三方物流查询；阶段目标是让后端数据库、API、状态流转先完整可用。

## 2. 当前缺口分析

### 2.1 账号体系缺口

当前已有：

- APP 登录 Demo。
- 管理端登录 Demo。
- access token + refresh token。
- `/me`、`/refresh`、`/logout`。

当前缺少：

- APP 邮箱密码注册。
- APP 邮箱密码登录。
- 客户密码哈希保存。
- 邮箱唯一约束。
- 后台客户账号 CRUD。
- 后台管理员账号 CRUD。
- 角色权限控制。

### 2.2 仓库体系缺口

当前已有：

- 材料库存。
- 材料库存流水。
- 材料锁定、释放、消耗。
- 简单成品库存查询。

当前缺少：

- 仓库和库位。
- 按订单 / 打印任务追踪的成品库存件。
- 打印完成后的手动入库动作。
- 入库单。
- 出库单。
- 发货单。
- 一个订单多个快递单号。
- 批量出库。
- 订单生产完成度和发货状态联动。

## 3. 总体业务流程

### 3.1 上架商品订单流程

```text
客户下单
  -> 管理员确认收款
  -> 创建打印任务
  -> 创建排期
  -> 打印任务完成
  -> 管理员转移入库
  -> 系统检查订单全部商品是否已入库
  -> 订单变为待发货
  -> 管理员填写一个或多个快递单号
  -> 创建发货单 / 包裹
  -> 批量确认出库
  -> 订单变为已发货或部分发货
```

### 3.2 个性化定制订单流程

```text
客户注册 / 登录
  -> 上传切片文件
  -> 提交定制表单
  -> 管理员审核
  -> 管理员报价
  -> 客户确认报价
  -> 管理员确认收款
  -> 创建打印任务
  -> 创建排期
  -> 打印任务完成
  -> 管理员转移入库
  -> 订单变为待发货
  -> 填写快递单号
  -> 批量出库
  -> 订单变为已发货
```

## 4. 状态设计

### 4.1 客户账号状态

```text
active      正常
disabled    禁用
deleted     软删除
```

阶段1不建议物理删除客户账户，避免历史订单找不到客户。

### 4.2 后台账号状态

```text
active      正常
disabled    禁用
deleted     软删除
```

最后一个超级管理员不允许禁用或删除。

### 4.3 订单状态扩展

建议订单状态扩展为：

```text
submitted             已提交
quote_confirmed       已确认报价
payment_confirmed     已确认收款
scheduled             已排期
printing              生产中
partially_completed   部分生产完成
completed             生产完成
partially_inbound     部分入库
in_warehouse          已全部入库
ready_to_ship         待发货
partially_shipped     部分发货
shipping              发货中
shipped               已发货
cancelled             已取消
```

关键规则：

- `ready_to_ship` 不由人工随意设置，优先由系统在全部成品入库后自动更新。
- 一个订单允许拆成多个打印任务。
- 只有订单所有需要交付的打印任务都完成并入库，订单才能进入 `ready_to_ship`。
- 有多个快递单时，未全部出库前为 `partially_shipped` 或 `shipping`。
- 全部发货单确认出库后，订单进入 `shipped`。

### 4.4 打印任务状态补充

当前打印任务已有基础状态，建议补充仓库状态字段，而不是把所有状态塞进 `status`：

```text
warehouse_status:
not_required       不需要入库
pending_inbound    待入库
inbounded          已入库
outbounded         已出库
```

打印任务主状态仍保持：

```text
pending -> scheduled -> printing -> completed
```

当 `print_tasks.status = completed` 时，系统将 `warehouse_status` 改为 `pending_inbound`。

### 4.5 成品库存件状态

```text
available     可发货
reserved      已分配给发货单
outbound      已出库
shipped       已发货
cancelled     作废
```

### 4.6 发货单状态

```text
draft         草稿
ready         已填写快递单，待出库
outbounded    已出库
cancelled     已取消
```

### 4.7 出库单状态

```text
draft         草稿
confirmed     已确认出库
cancelled     已取消
```

## 6. API 设计路线

### 6.1 APP 账号接口

#### 注册

```text
POST /api/v1/app/auth/register
```

请求：

```json
{
  "email": "user@example.com",
  "password": "Password123!",
  "nickname": "客户"
}
```

规则：

- 邮箱必填且唯一。
- 密码最小 8 位。
- 密码使用 bcrypt 或 argon2 哈希。
- 注册成功后可以直接返回 token，也可以要求登录；阶段1建议直接返回 token。

#### 邮箱密码登录

```text
POST /api/v1/app/auth/login
```

请求改为：

```json
{
  "email": "user@example.com",
  "password": "Password123!"
}
```

兼容策略：

- 如果前端尚未接入，可以临时保留旧手机号验证码接口为 `/api/v1/app/auth/login-demo`。
- 正式 `/login` 使用邮箱密码。

### 6.7 发货单和快递单号

#### 创建订单发货单

```text
POST /api/v1/admin/orders/{order_id}/shipments
```

请求：

```json
{
  "receiver_name": "张三",
  "receiver_phone": "13800000000",
  "receiver_address": "上海市...",
  "packages": [
    {
      "carrier_code": "sf",
      "carrier_name": "顺丰",
      "tracking_no": "SF123456789"
    },
    {
      "carrier_code": "zto",
      "carrier_name": "中通",
      "tracking_no": "ZTO987654321"
    }
  ],
  "stock_item_ids": [1, 2, 3],
  "remark": "分两个包裹发出"
}
```

规则：

- 订单必须是 `ready_to_ship`、`in_warehouse` 或 `partially_shipped`。
- 快递单号可以一个或多个。
- 所选库存件必须属于该订单。
- 所选库存件必须是 `available`。
- 创建发货单后，库存件状态改为 `reserved`。

#### 查询订单发货单

```text
GET /api/v1/admin/orders/{order_id}/shipments
```

#### 修改发货单

```text
PATCH /api/v1/admin/shipments/{shipment_id}
```

#### 删除发货单

```text
DELETE /api/v1/admin/shipments/{shipment_id}
```

规则：

- 只有 `draft` 或 `ready` 状态可删除。
- 删除时释放库存件，从 `reserved` 改回 `available`。

### 6.8 批量出库

#### 创建批量出库单

```text
POST /api/v1/admin/warehouse/outbounds/batch
```

请求：

```json
{
  "shipment_ids": [1001, 1002, 1003],
  "remark": "今日下午批量发货"
}
```

规则：

- 发货单必须已填写快递单号。
- 发货单状态必须是 `ready`。
- 自动创建 `warehouse_outbound_records` 和 `warehouse_outbound_items`。
- 先生成草稿出库单，不立即扣减。

#### 确认出库

```text
POST /api/v1/admin/warehouse/outbounds/{outbound_id}/confirm
```

规则：

- 出库单必须是 `draft`。
- 将关联库存件状态改为 `outbound` 或 `shipped`。
- 将发货单状态改为 `outbounded`。
- 检查订单所有应发库存是否已出库：
  - 全部出库：订单改为 `shipped`。
  - 部分出库：订单改为 `partially_shipped`。

#### 出库单列表

```text
GET /api/v1/admin/warehouse/outbounds
GET /api/v1/admin/warehouse/outbounds/{outbound_id}
```

## 7. 核心事务规则

### 7.1 注册事务

1. 校验邮箱格式。
2. 查询邮箱是否已存在。
3. 哈希密码。
4. 创建 `users`。
5. 创建 refresh token。
6. 返回 access token 和 refresh token。

### 7.2 打印任务转移入库事务

1. 加锁读取 `print_tasks`。
2. 校验任务状态为 `completed`。
3. 校验未入库。
4. 创建入库记录。
5. 创建库存件。
6. 更新任务 `warehouse_status`。
7. 汇总订单下全部任务入库状态。
8. 如全部入库，更新订单 `status = ready_to_ship`。
9. 写操作日志。

### 7.3 创建发货单事务

1. 加锁读取订单。
2. 校验订单允许发货。
3. 校验快递单号非空。
4. 加锁读取库存件。
5. 校验库存件属于订单且状态为 `available`。
6. 创建 `shipments`。
7. 创建 `shipment_packages`。
8. 创建 `shipment_items`。
9. 库存件状态改为 `reserved`。
10. 订单状态改为 `shipping` 或保持 `ready_to_ship`。

### 7.4 批量确认出库事务

1. 加锁读取出库单。
2. 校验出库单状态为 `draft`。
3. 加锁读取出库明细和库存件。
4. 库存件状态从 `reserved` 改为 `shipped`。
5. 发货单状态改为 `outbounded`。
6. 出库单状态改为 `confirmed`。
7. 检查每个订单是否全部发货。
8. 更新订单为 `partially_shipped` 或 `shipped`。
9. 写操作日志。

### 第 1 步：账号体系

- 扩展 ORM 模型。
- 扩展 Pydantic Schema。
- 实现 APP 注册。
- 改造 APP 邮箱密码登录。
- 实现后台客户账号 CRUD。
- 实现后台管理员账号 CRUD。
- 增加基础角色校验。

验收：

- APP 能注册账户。
- APP 能邮箱密码登录拿 token。
- token 能访问 `/api/v1/app/auth/me`。
- 管理员能创建、禁用、重置客户密码。
- 超级管理员能创建、禁用、重置后台账号。

### 第 2 步：仓库基础资料

- 新增仓库表和库位表。
- 实现仓库 CRUD。
- 实现库位 CRUD。
- Dashboard 增加仓库待办数量。

验收：

- 管理端能创建仓库。
- 管理端能创建库位。
- 入库时能选择仓库和库位。

### 第 3 步：生产完成入库

- 打印任务完成后标记 `pending_inbound`。
- 实现单任务转移入库。
- 实现订单批量转移入库。
- 实现成品库存件查询。
- 实现入库记录查询。

验收：

- 完成的打印任务可以入库。
- 重复入库会返回 `409`。
- 所有任务入库后订单自动变为 `ready_to_ship`。

### 第 4 步：发货单和快递单

- 实现订单发货单创建。
- 支持一个订单多个包裹。
- 支持一个订单多个快递单号。
- 发货单创建后锁定库存件。

验收：

- 订单待发货后可以填写快递单。
- 一个订单能保存多个快递单号。
- 被发货单占用的库存件不能再次被其他发货单选择。

### 第 5 步：批量出库

- 实现批量创建出库单。
- 实现确认出库。
- 出库后更新库存件、发货单、订单状态。

验收：

- 可选择多个发货单批量生成出库单。
- 确认出库后库存件变为已发货。
- 全部出库后订单变为 `shipped`。
- 部分出库时订单变为 `partially_shipped`。

## 10. 安全与一致性要求

### 10.1 密码安全

- 禁止保存明文密码。
- 使用 bcrypt 或 argon2。
- 登录失败不要暴露“邮箱存在但密码错”这类细节。
- 后台重置密码后，应让旧 refresh token 失效。

### 10.2 权限要求

建议最小权限：

| 角色 | 权限 |
|---|---|
| `super_admin` | 所有权限，含后台账号管理 |
| `admin` | 大部分业务权限，不含删除超级管理员 |
| `production_manager` | 打印机、排期、打印任务、入库 |
| `warehouse_manager` | 仓库、入库、发货、出库 |
| `customer_service` | 客户、订单、定制审核、报价查看 |

### 10.3 幂等要求

必须支持 `Idempotency-Key` 的接口：

```text
POST /api/v1/app/auth/register
POST /api/v1/admin/print-tasks/{task_id}/transfer-to-warehouse
POST /api/v1/admin/orders/{order_id}/transfer-to-warehouse
POST /api/v1/admin/orders/{order_id}/shipments
POST /api/v1/admin/warehouse/outbounds/batch
POST /api/v1/admin/warehouse/outbounds/{outbound_id}/confirm
```

### 10.4 并发要求

以下操作必须在事务中加锁：

- 邮箱注册唯一性。
- 打印任务入库。
- 库存件分配到发货单。
- 批量确认出库。
- 订单状态自动更新。

SQL Server 建议使用：

```sql
WITH (UPDLOCK, HOLDLOCK)
```

## 11. API 完整性验收用例

### 11.1 邮箱注册登录

1. APP 注册邮箱账户。
2. APP 使用邮箱密码登录。
3. APP 获取 `/me`。
4. APP 刷新 token。
5. 后台禁用该客户。
6. 客户 token 再访问接口返回 `403`。

### 11.2 后台账号管理

1. 超级管理员创建仓库管理员。
2. 仓库管理员登录。
3. 仓库管理员访问仓库接口成功。
4. 仓库管理员访问后台账号管理接口失败。
5. 超级管理员禁用仓库管理员。
6. 仓库管理员再次访问接口失败。

### 11.3 生产入库

1. 创建订单。
2. 确认收款。
3. 创建打印任务。
4. 打印任务状态改为 `completed`。
5. 转移入库。
6. 查询库存件。
7. 所有任务入库后订单变为 `ready_to_ship`。

### 11.4 多快递单发货

1. 对待发货订单创建发货单。
2. 填写两个快递单号。
3. 选择多个库存件。
4. 创建发货单后库存件变为 `reserved`。
5. 查询订单发货单，能看到多个包裹。

### 11.5 批量出库

1. 选择多个 `ready` 发货单。
2. 批量创建出库单。
3. 确认出库。
4. 发货单变为 `outbounded`。
5. 库存件变为 `shipped`。
6. 订单变为 `shipped` 或 `partially_shipped`。

## 12. 推荐落地顺序

最稳妥的执行顺序：

1. 先做账号体系，因为后续仓库操作需要明确操作者和权限。
2. 再做仓库基础资料，因为入库必须选择仓库和库位。
3. 再做生产完成入库，让订单能进入 `ready_to_ship`。
4. 再做发货单和多个快递单号。
5. 最后做批量出库和订单发货状态联动。

这样做的好处是每一阶段都能单独验收，不会把账号、仓库、物流状态混成一团。
