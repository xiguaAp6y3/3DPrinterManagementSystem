# 后端数据库方案缺陷分析与完善方案

## 1. 结论摘要

当前方案方向基本正确：FastAPI 本地运行、Caddy 暴露 API、SQL Server 落库，阶段1聚焦上架商品、定制切片文件、人工审核、人工报价、人工收款确认、手动排期和人工维护打印机状态。

主要缺陷集中在落地细节：

- 缺少幂等机制，订单创建、报价确认、收款确认、排期创建容易因重复请求产生重复数据。
- 排期模型不够严谨，“以订单为单位排期”和“一单多个打印任务”之间存在结构冲突。
- 库存锁定只描述了表和触发器，没有明确并发锁、释放、消耗、失败损耗的事务边界。
- 文件上传暴露面较大，切片文件可能很大，且包含打印机参数，需要隔离存储、权限校验和下载签名。
- Caddy 暴露 API 后，缺少请求体大小、上传超时、真实 IP、日志、TLS 和速率限制策略。
- 状态机只列了状态，没有强制合法迁移，后端容易出现跨状态更新。
- 触发器边界仍需收紧，避免触发器和 ORM 同时维护字段导致递归、死锁或难排查问题。
- 管理端权限目前只有粗粒度 admin，后续操作日志和责任追踪不够。
- SQL Server 字段没有统一并发控制字段，排期和库存并发更新风险较高。

## 2. 必须优先修正的问题

### 2.1 排期模型需要拆成“排期单 + 排期项”

现有问题：

- 业务确认“排期以订单为单位”。
- 业务也确认“一笔订单允许拆成多个打印任务”。
- 现有 `production_schedules` 同时包含 `order_id`、`print_task_id`、`printer_id`，容易变成一条记录既代表订单排期，又代表打印任务排期。

风险：

- 一个订单多个打印任务时，无法清晰表达多个打印机、多个时间段。
- 订单整体延期和单个任务延期混在一起。
- 排期冲突校验粒度不清。

修正方案：

新增两个表：

```text
production_schedule_orders    订单级排期主表
production_schedule_items     打印任务级排期明细表
```

`production_schedule_orders`：

- `id`
- `schedule_no`
- `order_id`
- `status`
- `planned_start_at`
- `planned_end_at`
- `due_at`
- `priority`
- `created_by`
- `created_at`
- `updated_at`
- `row_version`

`production_schedule_items`：

- `id`
- `schedule_order_id`
- `print_task_id`
- `printer_id`
- `scheduled_start_at`
- `scheduled_end_at`
- `status`
- `sort_order`
- `created_at`
- `updated_at`
- `row_version`

规则：

- 订单是否已排期看 `production_schedule_orders`。
- 打印机时间占用看 `production_schedule_items`。
- 防重叠只对 `production_schedule_items` 生效。
- 一个订单可以有一个排期主表，多条排期明细。

阶段1如果想简化，也应把原 `production_schedules` 明确定义为“排期明细表”，再增加 `schedule_group_no` 或 `schedule_order_id`。

### 2.2 所有关键写接口必须支持幂等

现有问题：

- APP 网络抖动、Flutter 重试、Caddy 超时重放、用户连点，都可能重复创建订单、重复确认报价、重复锁库存。

必须新增：

```text
Idempotency-Key: 客户端生成的唯一键
```

建议新增表 `idempotency_keys`：

- `id`
- `scope`
- `idempotency_key`
- `request_hash`
- `response_body`
- `status_code`
- `resource_type`
- `resource_id`
- `created_by_user_id`
- `created_by_staff_id`
- `expires_at`
- `created_at`

必须幂等的接口：

- `POST /api/app/orders`
- `POST /api/app/custom-requests`
- `POST /api/app/quotes/{quote_id}/confirm`
- `POST /api/admin/orders/{order_id}/payment-confirm`
- `POST /api/admin/production-schedules`
- `POST /api/admin/print-tasks`
- `POST /api/admin/inventory/materials/{material_id}/stock-logs`

规则：

- 相同用户、相同接口、相同 `Idempotency-Key`，返回第一次结果。
- 相同 key 但请求体 hash 不同，返回 `409 IDEMPOTENCY_KEY_CONFLICT`。
- key 保存 24 到 72 小时即可。

### 2.3 库存并发必须由事务和行锁控制

现有问题：

- 只依赖触发器防止负库存不够。
- 多个管理员同时排期可能同时看到材料足够，最后超锁库存。

修正方案：

锁定材料库存时必须在同一事务中执行：

```sql
SELECT *
FROM dbo.materials WITH (UPDLOCK, HOLDLOCK)
WHERE id = @material_id;
```

然后检查：

```text
available_weight = stock_weight - reserved_weight
available_weight >= request_weight
```

再更新：

```text
reserved_weight = reserved_weight + request_weight
```

最后写入：

- `inventory_locks`
- `material_stock_logs`

释放和消耗也必须是显式事务：

- 释放：`reserved_weight -= weight`，lock 状态改 `released`。
- 消耗：`reserved_weight -= weight`，`stock_weight -= weight`，lock 状态改 `consumed`。
- 损耗：`stock_weight -= loss_weight`，写 `loss` 流水。

触发器只负责兜底：

- 防止 `stock_weight < 0`。
- 防止 `reserved_weight < 0`。
- 防止 `reserved_weight > stock_weight`。

### 2.4 文件上传方案需要提高安全等级

现有问题：

- 切片文件可能非常大。
- 文件中包含打印机参数。
- 如果直接暴露静态 URL，客户可能越权下载其他客户文件。
- `.zip` 可能出现压缩炸弹或嵌套恶意文件。

修正方案：

文件生命周期：

```text
uploading -> stored -> verified -> linked -> archived/deleted
```

新增字段：

- `storage_bucket`
- `storage_key`
- `public_url` 不建议阶段1使用。
- `sha256`
- `upload_status`
- `verified_at`
- `virus_scan_status`
- `owner_type`
- `owner_id`

文件访问规则：

- 不直接返回真实存储路径。
- 下载使用后端鉴权接口或短期签名 URL。
- 客户只能访问自己的文件。
- 管理员可访问所有文件，但必须记录操作日志。

Caddy / 后端必须设置：

- 单文件大小限制。
- 上传超时。
- 允许扩展名白名单。
- 文件名重新生成，禁止信任原始文件名。
- `.zip` 解压分析后置，阶段1不自动解压执行。

### 2.5 Caddy 暴露 API 需要补充生产级限制

现有问题：

- 只写了反向代理，没有写请求体大小、超时、日志、真实 IP、上传路径策略。

建议 Caddyfile 增强：

```caddyfile
api.example.com {
    encode gzip

    request_body {
        max_size 300MB
    }

    log {
        output file ./logs/access.log
        format json
    }

    header {
        Strict-Transport-Security "max-age=31536000;"
        X-Content-Type-Options "nosniff"
        X-Frame-Options "DENY"
        Referrer-Policy "strict-origin-when-cross-origin"
    }

    reverse_proxy 127.0.0.1:8000 {
        header_up X-Forwarded-Proto {scheme}
        header_up X-Forwarded-Host {host}
        transport http {
            read_timeout 10m
            write_timeout 10m
        }
    }
}
```

注意：

- Caddy 默认没有完善的应用级限流，登录、上传、验证码接口的限流应在 FastAPI 中实现。
- 如果部署在内网穿透或云代理后面，需要明确信任代理链，否则真实 IP 不可靠。

### 2.6 状态机必须在服务层强约束

现有问题：

- 文档列了状态，但没有声明哪些迁移合法。
- 如果允许任意 PATCH 状态，数据很快会不可控。

修正方案：

后端维护状态迁移表。

订单合法迁移：

```text
submitted -> reviewing
submitted -> payment_confirmed
reviewing -> cancelled
reviewing -> payment_confirmed
quote_confirmed -> payment_confirmed
payment_confirmed -> scheduled
scheduled -> printing
printing -> post_processing
printing -> quality_check
printing -> completed
post_processing -> quality_check
quality_check -> completed
任意未完成状态 -> cancelled，需要管理员权限和备注
```

定制合法迁移：

```text
submitted -> reviewing
reviewing -> need_more_info
need_more_info -> submitted
reviewing -> rejected
reviewing -> quote_pending
quote_pending -> quoted
quoted -> quote_confirmed
quote_confirmed -> payment_confirmed
payment_confirmed -> scheduled
```

打印任务合法迁移：

```text
pending -> scheduled
scheduled -> printing
printing -> paused
paused -> printing
printing -> completed
printing -> failed
scheduled -> cancelled
failed -> pending
```

所有非法迁移返回：

```text
409 INVALID_STATE_TRANSITION
```

### 2.7 触发器边界要更保守

现有问题：

- `AFTER UPDATE` 触发器如果再更新同表的 `updated_at`，在 SQL Server 中容易带来递归、嵌套触发器、ORM 行版本混乱。

修正方案：

- `updated_at` 优先由应用层统一维护。
- 数据库只用 DEFAULT 设置 `created_at`。
- 必做触发器只保留：
  - 防负库存。
  - 打印机状态日志。
  - 排期重叠兜底。
  - 关键状态审计。

不要在阶段1做“所有表通用 updated_at 触发器”。

## 3. 次优先但应尽快补充的问题

### 3.1 缺少乐观锁字段

建议核心表增加：

```text
row_version ROWVERSION
```

适用表：

- `orders`
- `custom_requests`
- `quotes`
- `printers`
- `print_tasks`
- `production_schedule_orders`
- `production_schedule_items`
- `materials`
- `inventory_locks`

用途：

- 管理后台多人编辑时检测并发覆盖。
- 排期拖拽调整时避免覆盖别人刚保存的计划。

### 3.2 缺少软删除和归档策略

建议：

- 商品、SKU、打印机、材料使用 `is_deleted`。
- 订单、定制、报价不允许软删除，只允许取消或归档。
- 文件可逻辑删除，但保留审计记录。

### 3.3 缺少后台权限细分

阶段1可以只有管理员，但表结构应预留：

- `roles`
- `permissions`
- `staff_role_bindings`

如果不想阶段1实现权限表，至少 `staff_users.role` 要保留：

- `admin`
- `sales_admin`
- `production_admin`
- `inventory_admin`
- `viewer`

### 3.4 缺少业务编号生成策略

不能依赖前端生成业务编号。

建议格式：

```text
OD202607070001
CR202607070001
QT202607070001
PT202607070001
SC202607070001
```

生成方式：

- 阶段1可由后端基于日期 + SQL Server sequence 生成。
- 高并发时不要用 `MAX(no) + 1`。

建议增加 sequence：

- `seq_order_no`
- `seq_custom_request_no`
- `seq_quote_no`
- `seq_print_task_no`
- `seq_schedule_no`

### 3.5 缺少 API 版本策略

建议从阶段1开始使用：

```text
/api/v1/app/*
/api/v1/admin/*
```

如果暂时不加版本，后续 Flutter 客户端升级会比较痛苦。

推荐修正为：

```text
/api/v1/app/products
/api/v1/admin/orders
```

### 3.6 缺少错误码清单

建议统一错误码：

- `AUTH_INVALID_TOKEN`
- `AUTH_FORBIDDEN`
- `RESOURCE_NOT_FOUND`
- `VALIDATION_ERROR`
- `INVALID_STATE_TRANSITION`
- `IDEMPOTENCY_KEY_CONFLICT`
- `INSUFFICIENT_MATERIAL_STOCK`
- `SCHEDULE_TIME_OVERLAP`
- `FILE_TOO_LARGE`
- `FILE_TYPE_NOT_ALLOWED`
- `FILE_ACCESS_DENIED`
- `ORDER_PAYMENT_NOT_CONFIRMED`

## 4. API 设计修正建议

### 4.1 路径统一加版本

原：

```text
/api/app/products
/api/admin/orders
```

建议：

```text
/api/v1/app/products
/api/v1/admin/orders
```

### 4.2 写接口统一支持幂等

所有关键 `POST` 接口要求 Header：

```text
Idempotency-Key: uuid
```

未传时：

- 对普通管理创建接口可允许。
- 对订单、报价确认、收款确认、排期、库存变更应返回 `400 IDEMPOTENCY_KEY_REQUIRED`。

### 4.3 订单创建需要拆分上架商品和定制订单语义

上架商品订单：

```text
POST /api/v1/app/orders/listed-product
```

定制订单建议由报价确认后生成，不建议客户直接创建：

```text
POST /api/v1/app/quotes/{quote_id}/confirm
```

原因：

- 定制必须先审核报价。
- 避免客户绕过报价直接创建订单。

### 4.4 管理员确认收款必须不可重复改变金额

收款确认接口只确认状态，不修改订单金额。

```text
POST /api/v1/admin/orders/{order_id}/payment-confirm
```

规则：

- 如果已确认，重复同幂等 key 返回原结果。
- 如果已确认但新请求不同 key，返回 `409 PAYMENT_ALREADY_CONFIRMED`。
- 金额以 `orders.total_amount` 为准。

### 4.5 排期创建应使用排期主表和明细

建议请求：

```json
{
  "order_id": 3001,
  "planned_start_at": "2026-07-10T10:00:00",
  "planned_end_at": "2026-07-10T18:00:00",
  "due_at": "2026-07-11T18:00:00",
  "priority": 10,
  "items": [
    {
      "print_task_id": 5001,
      "printer_id": 1,
      "scheduled_start_at": "2026-07-10T10:00:00",
      "scheduled_end_at": "2026-07-10T14:00:00"
    },
    {
      "print_task_id": 5002,
      "printer_id": 2,
      "scheduled_start_at": "2026-07-10T14:30:00",
      "scheduled_end_at": "2026-07-10T18:00:00"
    }
  ],
  "material_locks": [
    {
      "material_id": 1,
      "weight": 250.5
    }
  ]
}
```

## 5. 数据库设计修正建议

### 5.1 新增 `idempotency_keys`

字段：

- `id`
- `scope`
- `idempotency_key`
- `request_hash`
- `response_body`
- `status_code`
- `resource_type`
- `resource_id`
- `created_by_user_id`
- `created_by_staff_id`
- `expires_at`
- `created_at`

唯一索引：

```text
(scope, idempotency_key)
```

### 5.2 拆分排期表

用以下两个表替代或增强 `production_schedules`：

- `production_schedule_orders`
- `production_schedule_items`

防重叠触发器改到 `production_schedule_items`。

### 5.3 增加 `inventory_transactions`

现有 `material_stock_logs` 只覆盖材料，不够表达成品和在制品。

建议增加统一库存事务表：

- `id`
- `inventory_type`
- `item_id`
- `change_type`
- `quantity`
- `weight`
- `before_quantity`
- `after_quantity`
- `before_weight`
- `after_weight`
- `related_order_id`
- `related_task_id`
- `created_by`
- `created_at`

阶段1如果只做材料，也可以先保留 `material_stock_logs`，但命名会限制后续扩展。

### 5.4 文件表增加安全字段

`model_files` 增加：

- `storage_bucket`
- `storage_key`
- `sha256`
- `upload_status`
- `virus_scan_status`
- `owner_type`
- `owner_id`
- `deleted_at`

### 5.5 核心表增加 `row_version`

SQL Server：

```sql
row_version ROWVERSION NOT NULL
```

后端更新时带上原 row_version，避免覆盖。

## 6. 后端实现修正建议

### 6.1 服务层必须负责事务

不要在 route 中直接拼多个仓储操作。

建议服务：

- `OrderService.create_listed_product_order`
- `CustomRequestService.submit`
- `QuoteService.confirm_by_customer`
- `PaymentService.confirm_offline_payment`
- `ScheduleService.create_schedule_for_order`
- `InventoryService.lock_materials`
- `InventoryService.consume_materials`

### 6.2 每个服务方法明确事务边界

示例：

```text
ScheduleService.create_schedule_for_order
  begin transaction
    lock order row
    verify payment confirmed
    lock printer rows
    check schedule overlap
    lock material rows
    create schedule order
    create schedule items
    create inventory locks
    update reserved stock
    update order status
  commit
```

### 6.3 后端需要统一状态机模块

建议：

```text
app/domain/state_machine.py
```

集中维护：

- 订单状态迁移。
- 定制状态迁移。
- 打印任务状态迁移。
- 打印机状态迁移。

任何接口不能直接绕过状态机更新状态。

## 7. 完善后的阶段1最小闭环

### 7.1 上架商品闭环

1. 管理员创建商品、图片、SKU。
2. 管理员上架商品。
3. 客户创建上架商品订单，带 `Idempotency-Key`。
4. 管理员确认线下收款。
5. 管理员创建订单级排期和排期明细。
6. 系统锁定材料库存。
7. 管理员推进打印任务。
8. 打印完成后消耗材料并记录成品。

### 7.2 定制闭环

1. 客户上传切片文件。
2. 客户提交定制表单，带 `Idempotency-Key`。
3. 管理员审核。
4. 管理员报价。
5. 客户确认报价。
6. 系统创建定制订单。
7. 管理员确认线下收款。
8. 管理员创建订单级排期和排期明细。
9. 系统锁定材料库存。
10. 管理员推进打印任务。

## 8. 文档修正优先级

P0 必须立刻修正：

- API 加版本。
- 关键写接口加幂等。
- 排期表拆分。
- 库存锁定事务补充行锁。
- 文件上传安全策略。
- 状态机合法迁移。

P1 建议阶段1实现：

- row_version 乐观锁。
- idempotency_keys 表。
- operation_logs 强化。
- Caddy 上传大小和超时配置。
- 统一错误码。

P2 后续增强：

- RBAC 权限表。
- 文件病毒扫描。
- 分片上传。
- 在线支付。
- 物流接口。
- 打印机自动接入。

## 9. 验收标准更新

方案完善后，阶段1后端验收应新增：

- 重复提交订单不会生成重复订单。
- 重复确认报价不会生成重复订单。
- 重复确认收款不会重复改变状态。
- 同一打印机重叠排期返回 `409 SCHEDULE_TIME_OVERLAP`。
- 材料库存并发锁定不会超卖。
- 非法状态迁移返回 `409 INVALID_STATE_TRANSITION`。
- 客户无法下载其他客户文件。
- Caddy 暴露的大文件上传能正常经过后端校验。
- 所有关键后台操作能追踪到管理员。
