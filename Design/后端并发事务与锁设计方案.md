# 后端并发事务与锁设计方案

## 1. 设计结论

本系统不建议把“多线程”作为保证数据安全的主要手段。FastAPI、SQL Server 和 Caddy 的并发能力应该分层设计：

- Caddy 负责连接、TLS、上传限制、超时和基础限流。
- Uvicorn / FastAPI 负责 HTTP 并发处理。
- Python 线程池只处理少量阻塞 I/O 或后台轻任务，不用于共享修改业务状态。
- SQL Server 事务、行级锁、唯一约束、检查约束、`rowversion`、幂等表负责数据一致性。
- 耗时任务用任务队列或后台 worker 解耦，不在请求线程里长时间持有数据库事务。

核心原则：

- 请求可以并发，关键业务写入必须串行化到正确的数据粒度。
- 事务必须短，锁必须小，锁顺序必须固定。
- 先做幂等，再做状态检查，再锁资源，再更新数据。
- 不允许通过 Python 全局变量、内存锁、单进程线程锁保证库存、订单、发货等核心数据一致性。

## 2. 当前方案缺陷

### 2.1 把多线程等同于提速是不准确的

FastAPI 本身已经能并发处理请求。对于当前同步 SQLAlchemy + pyodbc 的模式，性能瓶颈通常在：

- SQL 查询和事务时间。
- 数据库连接池大小。
- 文件上传和下载。
- 大量列表查询缺少索引。
- 锁等待和死锁。
- 请求内做了过多串行业务。

盲目给业务代码加线程可能带来：

- 同一个 SQLAlchemy Session 被多个线程使用，导致不可预测错误。
- 多线程同时更新库存、订单状态，造成竞态。
- 线程数超过数据库连接池，反而增加等待。
- 锁范围扩大，吞吐下降。

因此，提速优先级应为：

1. 数据库索引和查询优化。
2. 缩短事务时间。
3. 正确配置连接池和 Uvicorn worker。
4. 用后台任务处理非即时操作。
5. 对热点资源使用精确锁和幂等控制。

### 2.2 当前事务边界仍需服务层统一

部分接口已经能工作，但很多事务逻辑直接写在路由函数里。问题是：

- 事务边界不统一。
- 同类状态机可能在不同文件重复实现。
- `autoflush=False` 时，如果统计前没有 `flush`，会读到旧状态。
- 异常回滚、死锁重试、幂等响应保存不容易统一。

后续应逐步把关键写接口迁移到 service 层：

```text
api route -> service method -> repository/query helper -> db transaction
```

路由层只负责：

- 参数校验。
- 身份认证。
- 调用 service。
- 返回响应。

service 层负责：

- 幂等处理。
- 开启和提交事务。
- 加锁。
- 状态机校验。
- 写操作日志。
- 触发后台任务。

## 3. 运行时并发模型

### 3.1 推荐部署模型

阶段1建议：

```text
Caddy
  -> Uvicorn worker 1-2 个
      -> FastAPI
          -> SQLAlchemy connection pool
          -> SQL Server
```

Windows 本地开发：

```powershell
uvicorn app.main:app --host 127.0.0.1 --port 5000
```

生产或云服务器可逐步升级：

```powershell
uvicorn app.main:app --host 127.0.0.1 --port 5000 --workers 2
```

注意：

- Windows 上多 worker 可用性需要实际验证，部署脚本中要明确。
- worker 数不要大于数据库连接池能承受的并发。
- 上传大文件时不要让 worker 长时间阻塞业务写接口。

### 3.2 数据库连接池建议

当前可从小配置开始：

```python
create_engine(
    settings.database_url,
    pool_pre_ping=True,
    pool_size=5,
    max_overflow=10,
    pool_timeout=30,
)
```

估算方式：

```text
最大数据库连接数 ~= worker_count * (pool_size + max_overflow)
```

如果 2 个 worker，`pool_size=5`、`max_overflow=10`，峰值可能到 30 个连接。

阶段1建议：

- 本地开发：1 worker，`pool_size=5`。
- 小规模公网测试：1-2 worker，最大连接控制在 20-30。
- 不要先把 worker 和线程数拉高，先观察 SQL Server CPU、锁等待、连接数。

### 3.3 Python 多线程使用边界

允许使用线程或后台任务的场景：

- 图片缩略图生成。
- 文件病毒扫描或文件格式检查。
- 发送通知。
- 生成导出报表。
- 刷新统计缓存。
- 设备状态轮询。

不允许用 Python 线程锁保证的场景：

- 库存扣减。
- 材料锁定。
- 创建订单。
- 确认收款。
- 创建排期。
- 入库。
- 发货单预留库存。
- 出库确认。

原因：

- 多 worker 或多机器部署时，Python 内存锁只在单进程内有效。
- 系统重启后锁消失。
- 数据库才是最终一致性边界。

## 4. SQL Server 事务与锁策略

### 4.1 默认隔离级别

阶段1建议使用 SQL Server 默认 `READ COMMITTED`，关键写操作通过显式锁提升局部一致性。

不建议全局使用 `SERIALIZABLE`：

- 容易扩大锁范围。
- 列表查询和统计查询可能被阻塞。
- 高并发下死锁概率上升。

关键资源使用以下方式：

- 库存行：`UPDLOCK, ROWLOCK, HOLDLOCK`。
- 排期冲突检查：事务内锁定同一打印机的时间窗口。
- 订单状态更新：锁定订单行。
- 幂等键：唯一索引 + 插入抢占。
- 跨表业务唯一动作：必要时用 `sp_getapplock`。

### 4.2 行锁模板

材料库存锁定：

```sql
SELECT id, stock_weight, reserved_weight
FROM dbo.materials WITH (UPDLOCK, ROWLOCK, HOLDLOCK)
WHERE id = @material_id;
```

成品库存件发货预留：

```sql
SELECT id, status, order_id
FROM dbo.warehouse_stock_items WITH (UPDLOCK, ROWLOCK, HOLDLOCK)
WHERE id IN (@stock_item_ids);
```

订单状态迁移：

```sql
SELECT id, status, payment_status
FROM dbo.orders WITH (UPDLOCK, ROWLOCK)
WHERE id = @order_id;
```

排期冲突检查：

```sql
SELECT id
FROM dbo.production_schedule_items WITH (UPDLOCK, HOLDLOCK)
WHERE printer_id = @printer_id
  AND status NOT IN ('cancelled', 'completed')
  AND scheduled_start_at < @new_end_at
  AND scheduled_end_at > @new_start_at;
```

说明：

- `UPDLOCK` 防止多个事务同时读到可更新状态。
- `ROWLOCK` 倾向行锁，但 SQL Server 仍可能根据压力升级锁。
- `HOLDLOCK` 接近当前语句范围内的 serializable 语义，适合库存和排期窗口。

### 4.3 应用锁使用场景

对“同一业务资源只能同时执行一个动作”的场景，可以使用 SQL Server `sp_getapplock`：

```sql
EXEC sp_getapplock
    @Resource = @resource_name,
    @LockMode = 'Exclusive',
    @LockOwner = 'Transaction',
    @LockTimeout = 5000;
```

建议资源名：

```text
order:{order_id}:payment_confirm
order:{order_id}:schedule
order:{order_id}:shipment
order:{order_id}:outbound_confirm
printer:{printer_id}:schedule
material:{material_id}:reserve
```

使用原则：

- 只在事务内使用。
- 超时返回明确错误：`LOCK_TIMEOUT`。
- 不要对全局资源加锁。
- 不要用应用锁包裹长耗时任务。

### 4.4 乐观锁

后台管理端多人编辑商品、SKU、订单备注、打印机资料时，应使用 `row_version`：

请求时返回：

```json
{
  "id": 1,
  "name": "PLA 黑色",
  "row_version": "base64..."
}
```

更新时提交：

```json
{
  "name": "PLA 黑色 1kg",
  "row_version": "base64..."
}
```

后端更新：

```sql
UPDATE dbo.materials
SET name = @name
WHERE id = @id
  AND row_version = @row_version;
```

如果影响行数为 0：

```text
409 CONFLICT
错误码：ROW_VERSION_CONFLICT
提示：数据已被其他人修改，请刷新后重试
```

适合乐观锁的表：

- `products`
- `product_skus`
- `materials`
- `printers`
- `orders`
- `custom_requests`
- `production_schedule_orders`
- `production_schedule_items`

不适合只靠乐观锁的场景：

- 库存扣减。
- 发货库存件预留。
- 出库确认。

这些必须使用事务 + 行锁。

## 5. 统一事务管理方案

### 5.1 事务上下文

建议新增统一 helper：

```python
from contextlib import contextmanager

@contextmanager
def transaction(db: Session):
    try:
        yield
        db.commit()
    except Exception:
        db.rollback()
        raise
```

关键写 service 示例：

```python
def confirm_outbound(db: Session, outbound_id: int, staff_user_id: int, idempotency_key: str):
    with transaction(db):
        idem = idempotency.begin(db, scope="admin.confirm_outbound", key=idempotency_key)
        if idem.replayed:
            return idem.response_body

        outbound = lock_outbound(db, outbound_id)
        validate_outbound_can_confirm(outbound)
        mark_stock_items_shipped(db, outbound)
        sync_order_shipping_status(db, outbound.order_id)
        response = serialize_outbound(db, outbound)

        idempotency.save_success(db, idem, response)
        return response
```

### 5.2 事务中禁止做的事

事务内禁止：

- 上传或下载大文件。
- 调用外部 HTTP 服务。
- 发送邮件、短信、Webhook。
- 生成图片缩略图。
- 长时间计算报价。
- 等待打印机或第三方设备返回。

事务内只做：

- 读取并锁定必要行。
- 校验状态。
- 写业务表。
- 写流水和日志。
- 保存待执行后台任务记录。

### 5.3 事务提交后的动作

事务提交后再做：

- 发送通知。
- 生成缩略图。
- 推送订单状态。
- 刷新缓存。
- 触发设备轮询。

阶段1可以用 FastAPI `BackgroundTasks` 做轻任务；阶段2建议接入 Celery / RQ / Dramatiq。

## 6. 幂等设计

### 6.1 幂等表

建议表名：

```text
idempotency_keys
```

字段：

```text
id bigint identity primary key
scope nvarchar(100) not null
idempotency_key nvarchar(100) not null
actor_type nvarchar(20) not null        -- app_user / staff_user / system
actor_id bigint not null
request_hash char(64) not null
status nvarchar(20) not null            -- processing / succeeded / failed
status_code int null
response_body nvarchar(max) null
resource_type nvarchar(100) null
resource_id bigint null
locked_until datetime2 null
expires_at datetime2 not null
created_at datetime2 not null default sysutcdatetime()
updated_at datetime2 not null default sysutcdatetime()
```

唯一索引：

```sql
CREATE UNIQUE INDEX UX_idempotency_keys_scope_actor_key
ON dbo.idempotency_keys(scope, actor_type, actor_id, idempotency_key);
```

### 6.2 幂等执行规则

流程：

1. 检查 `Idempotency-Key` 是否存在。
2. 计算请求体 `request_hash`。
3. 事务内尝试插入幂等记录，状态 `processing`。
4. 如果唯一索引冲突，读取已有记录。
5. 如果 hash 不同，返回 `409 IDEMPOTENCY_KEY_CONFLICT`。
6. 如果已有记录 `succeeded`，直接返回历史响应。
7. 如果已有记录 `processing` 且未超时，返回 `409 REQUEST_PROCESSING`。
8. 如果已有记录 `processing` 已超时，可以抢占或要求客户端稍后重试。
9. 业务成功后保存响应体和资源 ID。

必须实现幂等的接口：

- 创建上架商品订单。
- 创建定制需求。
- 客户确认报价。
- 管理员确认收款。
- 创建打印任务。
- 创建排期。
- 材料入库、释放、消耗、损耗。
- 成品转移入库。
- 创建发货单。
- 批量创建出库单。
- 确认出库。

## 7. 关键业务锁矩阵

| 业务动作 | 事务边界 | 必须锁定的数据 | 幂等 | 备注 |
| --- | --- | --- | --- | --- |
| 创建上架商品订单 | 单事务 | SKU 行、必要时商品行 | 是 | 阶段1下单后生产，不扣成品现货 |
| 创建定制需求 | 单事务 | 用户提交记录、文件记录 | 是 | 文件先上传，业务表再绑定 |
| 客户确认报价 | 单事务 | quote 行、custom_request 行、order 行 | 是 | 防止重复生成订单 |
| 管理员确认收款 | 单事务 | order 行 | 是 | 已确认重复请求返回成功 |
| 创建打印任务 | 单事务 | order 行、printer 行 | 是 | 防止重复拆任务 |
| 创建排期 | 单事务 | order 行、printer 时间窗口、材料库存行 | 是 | 最容易发生并发冲突 |
| 材料锁定 | 单事务 | material 行 | 是 | 使用 `UPDLOCK, HOLDLOCK` |
| 材料释放 | 单事务 | inventory_lock 行、material 行 | 是 | 防止重复释放 |
| 材料消耗 | 单事务 | inventory_lock 行、material 行 | 是 | 防止重复扣减 |
| 打印完成入库 | 单事务 | print_task 行、order 行、stock_item 新行 | 是 | 防止重复入库 |
| 创建发货单 | 单事务 | order 行、warehouse_stock_items 行 | 是 | 库存件 `available -> reserved` |
| 批量创建出库单 | 单事务 | shipment 行、stock_item 行 | 是 | 发货单必须 `ready` |
| 确认出库 | 单事务 | outbound 行、shipment 行、stock_item 行、order 行 | 是 | 库存件 `reserved -> shipped` |

## 8. 死锁与重试策略

SQL Server 常见死锁错误：

```text
1205 deadlock victim
```

建议：

- 对关键写事务增加最多 2 次重试。
- 重试只允许在幂等保护下执行。
- 每次重试使用指数退避：100ms、300ms。
- 记录死锁日志，包括接口、scope、资源 ID、幂等键。

固定锁顺序：

```text
idempotency_keys
-> orders
-> order_items
-> print_tasks / schedule
-> materials
-> warehouse_stock_items
-> shipments
-> outbound_records
-> logs
```

同一类资源按 ID 升序加锁：

```text
stock_item_ids.sort()
material_ids.sort()
printer_ids.sort()
```

不要在一个事务里先锁 A 后锁 B，另一个事务先锁 B 后锁 A。

## 9. 后台任务与队列设计

### 9.1 阶段1轻量方案

可以使用 FastAPI `BackgroundTasks`：

- 商品图片生成缩略图。
- 删除过期临时文件。
- 写非关键日志。
- 发送简单通知。

限制：

- 进程崩溃后任务可能丢失。
- 不适合关键业务动作。
- 不适合长时间任务。

### 9.2 阶段2可靠方案

建议接入 Redis + Celery / RQ / Dramatiq：

```text
FastAPI
  -> 提交任务记录到数据库
  -> 提交队列消息
Worker
  -> 读取任务
  -> 执行
  -> 更新任务状态
```

适合：

- 切片文件安全扫描。
- 大文件解析。
- 打印任务自动轮询。
- 报表导出。
- 自动低库存预警。
- 设备状态同步。

关键要求：

- 队列任务也必须幂等。
- worker 每个任务使用独立数据库 Session。
- 任务状态必须落库：`pending/running/succeeded/failed/retry`.
- 不要在 HTTP 请求事务未提交前让 worker 读取相关业务数据。

## 10. 数据库约束与索引要求

### 10.1 唯一约束

必须有唯一约束：

- `orders.order_no`
- `shipments.shipment_no`
- `warehouse_stock_items.stock_item_no`
- `warehouse_outbound_records.outbound_no`
- `idempotency_keys(scope, actor_type, actor_id, idempotency_key)`
- `users.email`
- `staff_users.username`
- `staff_users.email`

### 10.2 防负库存约束

材料表：

```sql
ALTER TABLE dbo.materials ADD CONSTRAINT CK_materials_stock_non_negative
CHECK (stock_weight >= 0 AND reserved_weight >= 0 AND reserved_weight <= stock_weight);
```

库存锁：

```sql
ALTER TABLE dbo.inventory_locks ADD CONSTRAINT CK_inventory_locks_quantity_positive
CHECK (quantity > 0);
```

成品库存件：

```sql
ALTER TABLE dbo.warehouse_stock_items ADD CONSTRAINT CK_warehouse_stock_items_quantity_positive
CHECK (quantity > 0);
```

### 10.3 热点索引

建议补充或确认：

```sql
CREATE INDEX IX_orders_status_created_at
ON dbo.orders(status, created_at DESC);

CREATE INDEX IX_print_tasks_order_status
ON dbo.print_tasks(order_id, status);

CREATE INDEX IX_schedule_items_printer_time
ON dbo.production_schedule_items(printer_id, scheduled_start_at, scheduled_end_at, status);

CREATE INDEX IX_stock_items_order_status
ON dbo.warehouse_stock_items(order_id, status);

CREATE INDEX IX_shipment_items_stock_item
ON dbo.shipment_items(stock_item_id);

CREATE INDEX IX_outbound_items_stock_item
ON dbo.warehouse_outbound_items(stock_item_id);
```

## 11. API 层并发安全约定

### 11.1 写接口 Header

关键写接口统一要求：

```http
Idempotency-Key: uuid-or-random-string
```

缺失返回：

```json
{
  "code": "IDEMPOTENCY_KEY_REQUIRED",
  "message": "关键写接口必须提供 Idempotency-Key"
}
```

重复但请求体不同返回：

```json
{
  "code": "IDEMPOTENCY_KEY_CONFLICT",
  "message": "相同 Idempotency-Key 的请求参数不一致"
}
```

资源锁等待超时返回：

```json
{
  "code": "LOCK_TIMEOUT",
  "message": "资源正在被其他操作处理，请稍后重试"
}
```

并发版本冲突返回：

```json
{
  "code": "ROW_VERSION_CONFLICT",
  "message": "数据已被其他人修改，请刷新后重试"
}
```

### 11.2 查询接口

查询接口不需要加业务锁。

要求：

- 使用分页。
- 默认最大 `page_size` 不超过 100。
- 对列表查询字段加索引。
- Dashboard 统计可以允许短时间不完全实时。
- 大统计后续走缓存或异步汇总表。

## 12. Caddy 与超时设计

Caddy 建议：

- 限制上传大小，尤其是切片文件。
- 管理端和 APP API 分别配置合理超时。
- 大文件上传接口超时可以更长，普通 API 保持较短。
- 记录 access log，保留真实 IP。

普通 API 建议：

```text
请求超时：30s
事务目标耗时：< 2s
```

文件上传建议：

```text
上传接口超时：按最大文件大小配置
上传完成后只保存元数据，不在请求内解析大文件
```

## 13. 测试计划

### 13.1 单元测试

覆盖：

- 幂等 key 重复提交。
- 同 key 不同 body 返回 409。
- row_version 冲突返回 409。
- 状态机非法迁移返回 409。

### 13.2 并发测试

必须测试：

- 20 个并发请求同时锁定同一种材料，不允许 `reserved_weight > stock_weight`。
- 20 个并发请求同时确认同一笔收款，只允许一次真实状态变更。
- 20 个并发请求同时创建同一订单发货单，不允许同一库存件被重复预留。
- 20 个并发请求同时确认同一出库单，只允许一次确认成功。
- 多个管理员同时给同一打印机排期，时间段不能重叠。

### 13.3 压测目标

阶段1建议目标：

- 普通列表查询 P95 < 500ms。
- 关键写接口 P95 < 1000ms。
- 文件上传接口单独统计。
- 并发 20-50 用户后台操作时无负库存、无重复出库、无重复订单。

## 14. 落地顺序

### 第1步：统一事务 helper

- 增加 `app/services/transactions.py`。
- 所有关键写接口逐步迁移到 service 层。
- 明确 commit / rollback 只在 service 边界执行。

### 第2步：幂等表和中间件/helper

- 新增 `idempotency_keys`。
- 新增 `idempotency_service`。
- 对订单、收款、入库、发货、出库先接入。

### 第3步：库存与发货行锁

- 材料锁定、释放、消耗加 `UPDLOCK`。
- 成品库存件预留和出库加 `UPDLOCK`。
- 固定多库存件按 ID 升序加锁。

### 第4步：乐观锁

- 重要管理表返回 `row_version`。
- 修改接口要求携带 `row_version`。
- 冲突返回 `ROW_VERSION_CONFLICT`。

### 第5步：死锁重试和日志

- 对 SQL Server 1205 增加有限重试。
- 记录锁等待和死锁信息。
- Dashboard 增加异常统计。

### 第6步：后台任务队列

- 阶段1先用 `BackgroundTasks`。
- 阶段2接入 Redis + worker。
- 文件扫描、缩略图、报表和设备轮询从 HTTP 请求中剥离。

## 15. 阶段1最小可执行标准

阶段1不需要一次性做完整分布式架构，但必须做到：

- 关键写接口都有 `Idempotency-Key`。
- 库存、入库、发货、出库必须在事务中完成。
- 库存件预留和出库确认必须加数据库行锁。
- 订单状态同步必须在写入库存状态后 `flush` 再统计。
- 管理端多人编辑核心资料至少用 `row_version` 防覆盖。
- 所有关键写事务不调用外部服务、不处理大文件。
- 并发测试不出现负库存、重复出库、重复订单。

这样可以在不提前复杂化架构的前提下，保证多用户访问时的速度和数据安全。
